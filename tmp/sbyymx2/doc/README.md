## 第一部分：代码工程说明

### 1. 设计定位：配置驱动条带 + 中间行判奖

与旧版 `game/sbyymx`（Redis 预设盘 + 动态概率选档）不同，本实现为：

1. **`game_json.go` 默认 JSON** + **Redis `GetRedisGameJson(8917)`** 覆盖：  
   - **推荐（对齐策划四套概率）**：`pay_table` + `weight_side_middle_row` / `weight_middle_middle_row` / `weight_side_upper_lower_row` / `weight_middle_upper_lower_row`（各含 `symbols` + `weights`，解析时归一化）+ `plain_wild_multipliers` / `plain_wild_multiplier_probs`（纯百搭 `100` 中奖时抽 ×2～×100）。  
   - **兼容**：仅配置 **`reels`**（3 列条带）时，按列随机起点取连续 3 格。  
2. 权重模式下 **每格独立抽样**；条带模式下每列一条带。  
3. **仅中间行**判奖；左右列不出百搭（权重/条带校验）；中列可 `100`、编码倍率格（`101+n`）、空白 `9`。  
4. 下注：`base × multiple × 10`；奖金：`bet × stepMultiplier / 10`；`stepMultiplier` = 赔付 ×（编码 `mid-100` 或纯百搭抽样倍数）。  
5. 场景 Redis 含 `specialMode` 等**预留字段**（特殊模式/狂欢二期）。

### 3. 符号与赔付（速查）

- 符号 **1–7**：普通符号；赔付见配置 `pay_table`（默认 `2,4,6,8,10,50,100`）。  
- **100**：百搭；**>100**：倍率百搭，倍率为 `id - 100`。  
- **0 / 9**：视为中间行左格无效（不中奖）。

- `game/game_map.go`：**`NewGameBetMap[8917]`** → `sbyymx2.NewGame()`（走 `BetOrderExcute` 的 **Protobuf** 分支，与 hcsqy 同类）

### 二、盘面与「3×1」含义

策划写 **「旋转区域 3×1」**，同时给出 **上下行 + 中间行** 的**四套权重表**（两边/中间 × 中间行/上下行）。

| 解读 | 说明 |
|------|------|
| **逻辑中奖** | **一条支付线 = 中间行三个格子**（左、中、右三轴各一格）。 |
| **表现层** | 常见为 **3 列×3 行可见窗口**：仅**中间行**参与判定；上下行用于滚轮滚动、停轮、假条带或「空位」表现（如余额不足盘型里「第一轴两个、中间空」）。 |

**坐标建议**（与代码 `row=1` 为中间行一致）：

- 行：`0`上、`1`中（线）、`2`下  
- 列：`0`左、`1`中、`2`右  
- 支付线：`[1,0],[1,1],[1,2]`（行优先存盘时 index 为 3,4,5）

### 三、符号与赔付

#### 3.1 基础符号与赔付值

| 符号 | 赔付值 |
|------|--------|
| 羽毛 | 2 |
| 龙鱼 | 4 |
| 棕榈树 | 6 |
| 巴西铃鼓 | 8 |
| 沙锤 | 10 |
| 古典吉他 | 50 |
| 鼓 | 100 |

#### 3.2 百搭规则

1. 可替代任意基础符号（策划未写夺宝；若以后加夺宝需单独约定是否可代）。  
2. **只出现在中间滚轴**（中间列）；权重表里「中间出现」列含百搭/空白。  
3. 可带 **×2～×100** 的倍数（见第五节）。

#### 3.3 符号概率（策划表）

四组分布（**两边/中间 × 中间行/上下行**）。**当前代码**在 `game_json.go` / Redis JSON 中用 `weight_*` 四张表 + `pickSymbolForCell(row,col)` **逐格抽样**实现；条带模式 `reels` 为兼容旧配置的备选。

**与策划表一处需人工确认**：你方表格里「**两边出现权重（中间行）**」行末若列了「百搭/空白」权重，会与「**百搭只出现在中间滚轴**」冲突。实现上 **左右列中间行权重表仅允许符号 1～7**（校验禁止 `>=100`），与百搭规则一致；若策划最终要求左右也出百搭，需改规则与校验。

**文档笔误**：倍数表 **`x0.1`** → **0.1（10%）** 表示 **×10**。

### 四、投注与奖励公式

| 项 | 策划 |
|----|------|
| 投注大小档位 | 0.05 / 0.50 / 2.00（4 档中的 3 档写法，以表为准） |
| 倍数档位 | 1～10 |
| **基础投注** | **10** |
| 单注金额 | 与表一致：如 0.05×10×k = 0.5～5.0 等 |
| **奖励（基础）** | **奖励 = 投注大小 × 投注倍数 × 赔付值** |

**注意**：若还有**百搭额外倍数**或**倍数模式最终倍数**，策划表述为在基础奖上再乘（文档「中奖数字变为最终奖励」）；需与程序 **`stepMultiplier` 是否已含全部乘积** 对齐。

**加减档**：列出 0.50～200 等，供 UI 或快捷调整使用。

**初始下注金额**：策划写 **1**（需明确是「倍数」还是「某展示单位」——通常指默认档位组合下的体验值）。

### 五、倍数（百搭 ×2～×100）

#### 5.1 档位与概率（策划表）

| 倍数 | ×2 | ×3 | ×5 | ×10 | ×20 | ×50 | ×100 |
|------|-----|-----|-----|------|------|------|-------|
| 概率 | 0.3 | 0.3 | 0.2 | **0.1**（原文 x0.1） | 0.06 | 0.03 | 0.01 |

#### 5.2 策划给的「占比」语义（预估值）

- **中奖中的占比 0.5**：可能指「在中奖结果里，有多大比例会进入带倍数流程」。  
- **所有投注占比 0.17**：全量 spin 中进入某类倍数体验的比例。  
- **特殊模式相关**：另有「中奖中 0.25、倍数中 0.5、全投注 0.085」等，与**狂欢/特殊模式**联动；并写「触发动画后不进入特殊模式概率同等」——**非精确 RTP**，实现时可配置化。

**与代码（已实现）**：  
- 中间格为 **编码倍率百搭**（`id > 100`）时，倍率 = **`id - 100`**（2～100）。  
- 中间格为 **纯百搭 `100`** 且形成中奖线时，按配置 **`plain_wild_multipliers` + `plain_wild_multiplier_probs`** 抽取 ×2～×100（默认即上表）。  
- 条带模式（仅 `reels`、无四套权重）下，纯百搭仍走上述抽样；编码格仍用 `id-100`。

### 六、玩法流程（策划视角）

#### 6.1 基础局

1. 三轴停轮 → **仅中间行**判定是否同一基础符号或百搭代成功 → 得**赔付值**。  
2. 若有**百搭带倍** → 进入**倍数模式**（以下多为**客户端演出**，服务端给**最终倍率/最终奖额**即可）。  
3. **符号变大**（演出）：  
   - **乐器线**：中间百搭中奖 + 两边为乐器 → 中列百搭与两边乐器**拉满整列**、乐器 BGM、敲打/弹奏动画、中间跳舞。  
   - **自然线**：仅**中间百搭拉满**，**两边不变大**。  

#### 6.2 倍数模式（演出向）

1. 触发条件：中间列百搭中奖且存在倍数需求。  
2. 乐器：先变大再倍数滚动；自然：百搭列变大后走倍数。  
3. UI：约 **3 秒**滚动（快→慢→停）→ 分身飞到中奖数字 → 更新为**最终奖励**。  

**服务端**：通常只需保证 **`最终倍数` 与 `最终金额`** 与随机/预设一致；动画时长不必在服务端模拟。

#### 6.3 特殊模式（狂欢 / 「旋转到赢」）

| 步骤 | 策划 |
|------|------|
| 触发 | 普通下注后，**概率**触发（文档写「滚动停止前」为表现时机） |
| 表现 | 中间百搭出现并放大；倍数区与倍数模式同位置；顶部文案「旋转到赢，X2～X100」 |
| 机制 | 中间倍数与 **1、3 列**同转同停，约 **2 秒/次** |
| 结果 | **先 2～5 次无中奖转动，再必然中奖** |
| 其它 | 跳舞、背景动画 |

**与代码（二期）**：`SpinSceneData` 已预留 `specialMode` / `specialSpinRemain` 等 JSON 字段；**尚未**实现「2～5 次无奖再必中」的状态机与多步 `BetOrder`。需与网关 **proto** 扩展（如 `phase`、`next`）联调后再接。

### 九、落地建议（联调 / 二期）

| 投注大小 0.05 / 0.50 / 2.00；写「共 4 档」与列举 3 个数不一致 | 代码 **不枚举档位**；`BetOrderReq.BaseMoney` / `Multiple` 由网关与 **game_setting** 约束。 |
| 倍数 1～10 | 同上，由请求传入。 |
| **基础投注 10** | ✅ `_baseMultiplier = 10`。 |
| 单注金额表 | 与 `betAmount = base × multiple × 10` 一致。 |
| 加减档、初始下注 1 | **UI/配置**；服务端只校验 `betAmount > 0`。 |

#### 10.9 奖励计算（策划公式 ↔ 程序）

策划：**奖励 = 投注大小 × 投注倍数 × 赔付值**（并可在百搭倍数上乘）。

程序：

- 单注：`betAmount = base × multiple × 10`（`updateBetAmount`）。
- 派彩：`bonus = betAmount × stepMultiplier / 10`（`updateBonusAmount`）。
- `stepMultiplier` = **赔付表数值** × **百搭倍**（编码 `mid-100` 或纯百搭抽样）（`processWinInfos`）。

代数：`bonus = base × multiple × stepMultiplier`，与「投注 × 倍数 ×（赔付×百搭倍）」一致；**基数 10 在 bet 与 bonus 中各出现一次，相消后不再单独出现在最终乘积里**。

#### 10.14 综合差异与风险清单（执行用）

1. **特殊模式 / 狂欢**：完全未做；预留 scene 字段未驱动逻辑。  
2. **条件倍数占比**（0.5 / 0.17 等）：未做；纯百搭线必抽 `plain_wild_*`。  
3. **初始/余额不足 teach 盘**：未做；客户端自备。  
4. **乐器/自然**：未在 PB 标注；客户端用符号 ID 推断或二期加字段。  
5. **左右中间行权重表是否含百搭**：以「仅中轴百搭」为准已实现；若策划改表需同步改校验。  
6. **三百搭线**（若左右错误出现 100）：当前 **不赔付**（左右必须 1～7）；与条带/权重校验一致。
