# Claude Code 全局开发指南

适用于所有项目。语言/框架规范见各项目 `CLAUDE.md`。

---

## 🎯 工作规则（AI 必须遵守）

1. **先方案后代码**：写任何代码前，必须输出实现方案，等待批准
2. **需求澄清**：需求模糊时，必须先提出 2-3 个具体问题，再动手
3. **任务拆分**：单次任务涉及超过 3 个文件时，必须拆分成子任务，逐一向用户确认
4. **测试先行**：修复 bug 前，必须先编写能复现该 bug 的测试用例
5. **边缘案例**：完成代码后，必须列出至少 2 个边缘场景，并说明如何测试
6. **复盘改进**：每次被纠正后，必须简要说明错误原因，并提出避免再犯的具体方法
7. **重构三连**：重构完成后必须依次执行：①精简冗余代码 ②运行 code-review 检查 ③确认编译通过

## 🔨 代码规范（AI 生成代码时必须遵守）

### 命名规范
- 函数/类型命名必须表达用途，禁止单字母命名（循环变量除外）
- 同一概念在整个项目中必须用相同词汇

### 函数设计
- 函数长度 ≤50 行，超过必须拆分
- 嵌套深度 ≤3 层，超过必须用提前返回重构
- 参数数量 ≤3 个，超过必须封装为结构体

### 注释规范
- 所有导出函数必须有注释，格式：`// FunctionName does something`
- 注释必须说明“为什么这么做”，而非“代码在做什么”
- 复杂算法必须附简要说明

### 错误处理
- 所有错误必须被处理或显式忽略（忽略时必须注释理由）
- 错误信息必须小写开头，无标点结尾

### 安全性
- 禁止 SQL 字符串拼接
- 禁止命令注入（`os/exec` 输入必须验证）
- 禁止硬编码密钥/密码
- 禁止 `InsecureSkipVerify: true`

## 🔧 代码修改策略

AI 修改代码时必须遵循以下优先级：

1. 优先修复 bug
2. 保持现有行为不变
3. 最小修改原则（只修改必要代码）
4. 不修改无关代码
5. 不进行未被要求的重构

### 函数定义顺序
- **被调用函数在前**：f1 调用 f2，则 f2 定义在 f1 之前
- **阅读顺序**：从上到下，先看到底层细节，再看到上层逻辑

```go
// ✅ 正确
func validateInput(input string) error {    // 被调用者在前
    return nil
}

func processInput(input string) error {     // 调用者在后
    return validateInput(input)              // 调用上面的函数
}

// ❌ 错误
func processInput(input string) error {     // 调用者在前
    return validateInput(input)
}

func validateInput(input string) error {    // 被调用者在后
    return nil
}```

## ✅ 代码完成自查清单

AI 完成代码后，必须逐项核对并报告结果：

- [ ] 所有函数 ≤50 行，嵌套 ≤3 层
- [ ] 所有导出函数有注释
- [ ] 所有错误被处理或显式忽略（已注明理由）
- [ ] 未修改无关代码
- [ ] 修改保持最小变更原则
- [ ] 无 SQL 拼接
- [ ] 无命令注入风险
- [ ] 无硬编码密钥
- [ ] 无不安全 TLS（`InsecureSkipVerify: true`） （可以手动询问是否需要）
- [ ] 已列出边缘场景及测试方法
- [ ] 已运行 code-review 检查
- [ ] 编译通过

## ⚡ 性能优化原则

- **可读性优先**：性能优化必须基于实际性能数据
- **验证驱动**：任何优化需有基准测试支撑，提升 ≥20% 才引入复杂度

## 🌐 语言交互规范

1. **默认语言**：与用户的所有对话必须使用中文
2. **技术术语**：保留英文代码标识符和技术术语，但解释和沟通使用中文
3. **分析结果**：所有代码分析、审查结果和报告必须以中文呈现
4. **错误信息**：将英文错误信息翻译为中文向用户提供说明
